# T1.3 — Ingest Pipeline: How It Works

> **Audience:** beginner to intermediate. Assumes you know what TypeScript is
> but not necessarily Drizzle ORM, BullMQ, or Fastify.

---

## 0. Principle #1 compliance

Every part of this pipeline operates **exclusively on saved `.w3g` replay files**
uploaded after the game ends. There is no live-game data path, no process-memory
reader, no packet sniffer, and no overlay. This is a hard invariant enforced both
in code comments and in the CI build — see CLAUDE.md §Principles.

---

## 1. Big picture: the upload-to-query flow

```
Developer / UI
     │
     │  POST /replays   (multipart, .w3g file)
     ▼
┌──────────────────────────────────────────────────────────┐
│  apps/api-node/src/server.ts  (Fastify HTTP server)      │
│                                                          │
│  1. Buffer the upload                                    │
│  2. SHA-256 hash the bytes  ──────► dedup check          │
│  3. createPendingReplay()   ──────► INSERT replays row   │
│     (already exists?)       ◄────── return existing id   │
│  4. writeFile() ── save <sha256>.w3g to uploads/         │
│  5. queue.add() ─────────────────────────────────────┐   │
│                                                      │   │
│  Return 202 { replayId, status:"pending" }           │   │
└──────────────────────────────────────────────────────┼───┘
                                                       │
                                              BullMQ job
                                         { replayId, filePath }
                                                       │
                                                       ▼
┌──────────────────────────────────────────────────────────┐
│  apps/api-node/src/worker.ts  (BullMQ Worker)            │
│                                                          │
│  6.  setReplayStatus(db, id, 'parsing')                  │
│  7.  parseReplayFile(filePath)  ◄── @wc3-coach/parser    │
│          └── returns ReplayTimeline                      │
│  8.  persistTimeline(db, timeline) ◄── @wc3-coach/db     │
│          ├── upsert patch_versions                       │
│          ├── UPDATE replays row (status='done')          │
│          ├── INSERT replay_players                       │
│          └── bulk INSERT game_events (chunks of 1 000)   │
│  9.  (failure) setReplayStatus(db, id, 'error', msg)     │
│                └── BullMQ retries (max 3, exponential)   │
└──────────────────────────────────────────────────────────┘
                                                       │
Developer / UI                                         │
     │                                                 │
     │  GET /replays/:id                               │
     ▼                                                 │
┌──────────────────────────────────────────────────────────┐
│  apps/api-node/src/server.ts                             │
│                                                          │
│  getReplayWithTimeline(db, id)                           │
│  ├── replay.status !== 'done'  →  return status only     │
│  └── status === 'done'         →  return full timeline   │
│        players[] + events[]                              │
└──────────────────────────────────────────────────────────┘
```

## 2. File and module map

### `packages/db` — the DB layer

| File | Purpose |
|---|---|
| `src/schema.ts` | Drizzle ORM table definitions — the single source of truth for the DB shape |
| `src/client.ts` | `createDb(url)` factory; wraps a `pg.Pool` + Drizzle; exports row types |
| `src/map.ts` | **Pure** converters: `ReplayTimeline` → insert-row objects. No DB needed. Unit-tested. |
| `src/persist.ts` | Actual DB operations: `createPendingReplay`, `persistTimeline`, `setReplayStatus`, `getReplayWithTimeline` |
| `src/migrate.ts` | Standalone CLI script — applies pending migrations from `db/migrations/` |
| `drizzle.config.ts` | Drizzle Kit config — schema path, migration output dir, dialect |

**Why this split?** `map.ts` is pure so it can be unit-tested without a running
Postgres (no mocking needed). `persist.ts` contains the I/O and is tested at
the integration level once Docker is available.

### `apps/api-node` — the HTTP + worker process

| File | Purpose |
|---|---|
| `src/config.ts` | Read and validate all env vars at startup; fail fast if required vars are missing |
| `src/queue.ts` | BullMQ `Queue` instance, `IngestJobData` type, `pingRedis()` health helper |
| `src/server.ts` | Fastify app: `POST /replays`, `GET /replays/:id`, `GET /health` |
| `src/worker.ts` | BullMQ `Worker`: parse → persist lifecycle |
| `src/index.ts` | Package entry point; re-exports version constants |

Server and worker are **separate entry points** (`dist/server.js` and `dist/worker.js`)
so they can be run in separate processes or scaled independently. For local dev
a single terminal for each is fine; in production you'd run N workers.

## 3. The replay lifecycle state machine

```
                   POST /replays upload
                          │
                          ▼
                      [ pending ]
                          │
              worker picks up job (sets status)
                          │
                          ▼
                      [ parsing ]
                     /           \
              success              failure
                 │                    │
                 ▼                    ▼
             [ done ]            [ error ]
                                     │
                              BullMQ retry
                           (up to 3 attempts,
                            exponential backoff)
                                     │
                            still failing?
                                     │
                                 [ error ]
                              (inspect in BullMQ
                               failed job queue)
```

**Where each transition happens:**

| Transition | Code location |
|---|---|
| `→ pending` | `createPendingReplay()` in `packages/db/src/persist.ts`, called from `POST /replays` |
| `pending → parsing` | `setReplayStatus(db, id, 'parsing')` at start of worker job |
| `parsing → done` | `persistTimeline()` in `packages/db/src/persist.ts` — the UPDATE sets `status='done'` |
| `parsing → error` | `setReplayStatus(db, id, 'error', message)` in the worker catch block |

## 4. The database schema

Four tables for the ingest pipeline (T2.1 subset). Design doc §5.2 is the
authoritative reference.

### `patch_versions`

```
id          uuid PK  DEFAULT gen_random_uuid()
version     text     "2.00", "1.36.1"
build_number integer  6117
released_at timestamptz nullable
created_at  timestamptz DEFAULT now()
UNIQUE(version, build_number)
```

Exists so game data is tied to the correct balance state. A replay references
a patch via `replays.patch_id → patch_versions.id`. Stats and timings shift
between patches — this table is the anchor for that versioning.

### `replays`

```
id          uuid PK
file_hash   text UNIQUE   SHA-256 of .w3g bytes — dedup key
map_id      text nullable  "map:echo_isles"
played_at   timestamptz nullable  (not in raw .w3g — enrichment deferred)
duration_ms integer nullable
patch_id    uuid → patch_versions(id) nullable
winner_slot integer nullable  (null for FLO replays)
status      text  'pending'|'parsing'|'done'|'error'
error       text nullable
raw_meta    jsonb nullable  {patchId, playerCount, eventCount}
created_at  timestamptz
updated_at  timestamptz
```

`file_hash` is the deduplication key: uploading the same replay twice returns
the existing row instead of creating a duplicate and re-parsing.

`status` + `error` are the ingest lifecycle fields. `raw_meta` stores a small
summary so the GET endpoint can answer basic questions without joining other tables.

### `replay_players`

```
id          uuid PK
replay_id   uuid → replays(id) ON DELETE CASCADE
slot        integer   1-based player slot
player_name text
race_id     text nullable  "race:O"  (provisional until T2.2 ontology)
apm         integer nullable
result      text  'win'|'loss'|'unknown'  (branded to ReplayPlayer['result'])
UNIQUE(replay_id, slot)
```

One row per player per replay. `ON DELETE CASCADE` means deleting a replay
deletes its players automatically.

### `game_events`

```
id         bigint GENERATED ALWAYS AS IDENTITY PK  (NOT uuid — high volume)
replay_id  uuid → replays(id) ON DELETE CASCADE
slot       integer
t_ms       integer    milliseconds from game start
type       text       'build'|'train'|'upgrade'|...  (branded to GameEventType)
entity_ref text       "unit:opeo", "building:oalt"   (provisional until T2.2)
payload    jsonb      {fourcc, resolved, ...}
INDEX(replay_id)
INDEX(replay_id, slot)
INDEX(replay_id, type)
```

`bigint GENERATED ALWAYS AS IDENTITY` instead of UUID: a single replay can
have thousands of events and UUID primary keys add ~40 bytes per row plus
index fragmentation. The three indexes cover the common access patterns:
"all events for a replay", "one player's events", "events of a given type".

### How it maps to shared-types

```
ReplayTimeline     →  replays row + replay_players rows + game_events rows
ReplayPlayer       →  replay_players row
GameEvent          →  game_events row
GameEventType      →  game_events.type  (.$type<GameEventType>() brand)
```

The `.type<>()` brand in Drizzle means the TypeScript layer carries the correct
literal union — you get autocompletion and type errors for free without a separate
Zod schema.

## 5. Why BullMQ + Redis

**Parallelism.** The Ryzen 7 9700X has 16 threads. Parsing a `.w3g` file is
CPU-bound (JSON deserialization + event normalization). With `INGEST_CONCURRENCY=4`
you can parse 4 replays at once without starving the HTTP server or DB.

**Resilience.** If the worker crashes mid-parse (power cut, OOM), BullMQ's
retry mechanism (3 attempts, exponential backoff: 1s → 2s → 4s) will re-attempt
the job when the worker restarts. Without a queue the upload would be lost.

**Dedup.** The `jobId: row.id` option on `queue.add()` makes enqueue idempotent.
If the server receives the same file twice before the worker finishes the first
job, the second call just returns the existing row (the `alreadyExisted` flag) — no
duplicate jobs.

**Separation of concerns.** The HTTP server is non-blocking: `POST /replays`
returns 202 in milliseconds. The client polls `GET /replays/:id` for status.
This means you can drop 5 replays at once after a session and all 5 start
processing in parallel.

## 6. Why Drizzle ORM

Drizzle is a "SQL-first" ORM: you write the schema in TypeScript, it generates
the SQL, and the query builder returns types inferred directly from the schema —
no code generation step needed at query time, no runtime magic.

**Migration workflow:**

```
1. Edit packages/db/src/schema.ts
2. corepack pnpm --filter @wc3-coach/db db:generate
   → drizzle-kit compares schema.ts against the last migration
   → generates a new .sql file in db/migrations/
3. Review the .sql file (commit it to git)
4. corepack pnpm --filter @wc3-coach/db db:migrate
   → runs migrate.ts which applies pending .sql files
   → records which files were applied in __drizzle_migrations table
```

The SQL files in `db/migrations/` are the durable record of every schema change,
exactly like Flyway or Liquibase. You never hand-edit a live DB.

## 7. How to run it end-to-end locally

### Prerequisites

- Docker Desktop running
- `.env` file at repo root (copy from `.env.example`)
- A `.w3g` replay file to test with

### Step 1: start backing services

```bash
# From repo root
docker compose up -d
docker compose ps   # wait for postgres and redis to show "healthy"
```

### Step 2: run migrations

```bash
# In a new terminal, with .env loaded
cd C:\Work\wc3
# Load env (Git Bash / WSL):
source .env
# Or PowerShell:
Get-Content .env | ForEach-Object { if ($_ -match '^\s*([^#][^=]*)=(.*)$') { [System.Environment]::SetEnvironmentVariable($Matches[1].Trim(), $Matches[2].Trim()) } }

corepack pnpm --filter @wc3-coach/db db:migrate
# Output: [migrate] All migrations applied successfully.
```

### Step 3: build everything

```bash
corepack pnpm turbo build
```

### Step 4: start the worker (Terminal 1)

```bash
# Set DATABASE_URL and REDIS_URL in the shell first (or run via dotenv-cli).
node apps/api-node/dist/worker.js
# Output: [worker] Listening on queue="replay-ingest" concurrency=4
```

### Step 5: start the API server (Terminal 2)

```bash
node apps/api-node/dist/server.js
# Output: Server listening at http://0.0.0.0:8787
```

### Step 6: upload a replay

```bash
curl -X POST http://localhost:8787/replays \
  -F "replay=@/path/to/your.w3g"
# Returns: {"replayId":"...", "status":"pending", "deduplicated":false}
```

### Step 7: poll for completion

```bash
REPLAY_ID="<id from step 6>"
curl http://localhost:8787/replays/$REPLAY_ID
# While parsing: {"replayId":"...","status":"parsing"}
# When done:     {"replayId":"...","status":"done","players":[...],"events":[...]}
```

### Step 8: health check

```bash
curl http://localhost:8787/health
# {"ok":true,"checks":{"db":"ok","redis":"ok"}}
```

### Uploading the same file twice (dedup)

```bash
curl -X POST http://localhost:8787/replays -F "replay=@/path/to/your.w3g"
# Returns 200: {"replayId":"<same id>","status":"done","deduplicated":true}
```

## 8. Provisional and deferred items

| Item | Status | Future task |
|---|---|---|
| `replayId` in events | Currently = fileHash from parser; the ingest layer uses the DB uuid after persist | Working as designed |
| `playedAt` | Always null (w3gjs has no wall-clock timestamp) | T2.3 / FLO enrichment |
| `entity_ref` values | Provisional strings ("unit:opeo") not yet resolved to ontology IDs | T2.2 ontology import |
| `winner_slot` | Null for all FLO replays (winningTeamId = -1) | FLO enrichment or Observer API |
| `deaths / positions` | Not available in raw .w3g | T1.4 — Observer API on replay playback |
| `race_id` | Passed through from parser (e.g. "race:O"); not yet a FK into a races table | T2.2 ontology schema |

## 9. How the packages relate to each other

```
@wc3-coach/shared-types  ←── defines GameEvent, ReplayTimeline, GameEventType
       ▲                               ▲
       │                               │
@wc3-coach/parser  ────────────────────┘
  parseReplayFile() → ReplayTimeline
       │
       │  (BullMQ job)
       ▼
@wc3-coach/db
  schema.ts  ─── Drizzle tables
  map.ts     ─── pure ReplayTimeline → rows
  persist.ts ─── DB operations
       │
       ▼
  PostgreSQL 16
```

The shared-types package is the integration seam: both the parser (which produces
data) and the DB layer (which stores it) import from the same types. If you change
a type in shared-types, the TypeScript compiler will immediately flag any
inconsistency in either consumer.
